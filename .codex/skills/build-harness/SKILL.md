# build-harness

harness-factory의 구성자 수행 프로토콜(`docs/CONSTRUCTOR-PROTOCOL.md`)을 실행한다.

## 절차

1. **Phase 0 — 원칙 로드·자료 수집**: 이 레포의 `README.md`, `docs/CONSTRUCTOR-PROTOCOL.md`, `principles/` 전체(01~07), `interview/QUESTION-BANK.md`, `CHECKLIST.md`를 읽는다. 명령 시점 기준 대상의 읽을 수 있는 관련 자료(README·docs, 빌드/테스트/CI 설정, 기존 규칙 파일, 구조, 이력)를 최대한 수집해 (a) 결정적 evaluator 후보, (b) 질문 소거 재료, (c) 하네스 목적 가설을 만든다.
2. **Phase 1 — 인터뷰**: `interview/QUESTION-BANK.md`의 1차 배치 4문항을 런타임의 사용자 질의 수단(Codex: request_user_input 또는 일반 질의, Claude Code: AskUserQuestion 등)으로 질의한다 (코드베이스에서 이미 확인한 답은 소거). Q1에서는 Phase 0의 목적 가설을 제시하고 사용자 입력으로 확정받는다 — 목적은 기본값이 없다. 답변 분기에 따라 2차 배치를 최대 4문항 추가. 사용자가 "알아서"라고 하면 기본값 일괄표를 적용한다.
3. **Phase 2 — 결정 확정**: 답변을 템플릿 치환 필드에 매핑하고, 인터뷰 요약과 적용 기본값을 D-001 초안으로 정리한다. 설계 방향은 기본값(코스트 기반 자동검증·보완)을 적용하되, 사용자 요청·방향성이 다르면 사용자가 우선 — 오버라이드는 D-001에 근거와 함께 기록.
4. **Phase 3 — 인스턴스화**: `templates/` 이하를 대상 위치(기본 `<대상>/harness/`)로 복사하며 `{{...}}`를 전부 치환한다. README §4 산출물 계약의 구조·파일명을 유지한다. state.json에 초기 큐와 `next_action`을 채운다. 미사용 축은 파일 내 "미사용 — 사유" 표기. 생성 문서는 문서 규율(원칙 7: 인덱스 기반 100줄 기본값, 예외는 표기)을 따른다.
5. **Phase 4 — 검증·보완 루프(최대 3회)·인도**: `CHECKLIST.md` 전 항목 수행. 특히 `grep -rn '{{' <대상>` 0건 확인과 콜드스타트 테스트(HARNESS.md만 읽고 3문항에 답하기)를 실제로 수행한다. fail이 있으면 보완 후 재검증을 최대 3회 회전하고, 각 회전의 보완 내용을 D-001에 누적 기록한다. 3회 후 잔여 fail은 숨기지 말고 인도 보고에 명시한다. 인도 보고(생성 트리 / 적용 결정·설계 방향 / 검증 회전 결과 / 첫 실행 방법 / 첫 백로그)로 마친다.

## 인자 해석

- 경로가 주어지면: 해당 경로를 대상 프로젝트로 삼는다.
- 작업 설명이 주어지면: Q1(대상·산출물)의 답으로 선반영하고 인터뷰에서 재확인만 한다.
- 인자가 없으면: 현재 작업 디렉토리를 대상으로 Q1부터 질의한다.
