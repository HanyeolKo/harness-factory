# RECOVERY PLAYBOOK — 실패 대응

모든 실패는 **분류·등급 판정부터** 한다. 모든 실패는 두 값을 함께 받는다: **원인 분류** + **대응 등급**.

## 대응 등급 — 2단계

| 등급 | 의미 | 행동 |
|---|---|---|
| **R (재시도)** | 자동 회복 허용 | 분류별 대응을 상한 내에서 수행. 상한 도달 시 S로 승격 |
| **S (중지)** | 자동 회복 금지 — 사용자 피드백 필요 | 즉시 체크포인트 → 에스컬레이션 보고 → 대기. **사용자 피드백 없이 재개 금지** |

## 실패 분류표 (분류 → 기본 등급 → 대응)

| 분류 | 판별 신호 | 등급 | 대응 |
|---|---|---|---|
| 일시적 (transient) | git push 네트워크 오류 등 | R | 백오프 재시도: 3회, 2s/4s/8s |
| 구조적 (structural) | 개정이 상호참조를 깨뜨림, verify.sh 실행 불가 | R | 재시도 금지. 진단 → 수정 → EVAL 재실행. 같은 단위 수정 시도 2회 실패 시 S 승격 |
| 스코프 (scope) | 안건 정의 오류, 불변 조건 충돌 발견 | **S** | 즉시 중지 프로토콜 |
| 예산 (budget) | 단위/세션 예산 초과 | R | 체크포인트 → 분할 또는 세션 교체. 총예산 소진은 S |
| 게이트 (gate) | 산출물 계약·불변 조건 문구 변경 필요 | **S** | 승인 요청. 게이트는 실패가 아니라 설계된 중지점이다 |

## 승격 규칙 (R → S)

- 재시도/수정 시도 상한 도달
- 같은 실패 키가 다른 작업 단위에서 재발해 원인이 하네스 수준으로 의심될 때
- 분류 자체가 판정 불가할 때 (모르는 실패는 S다)

## 중지 프로토콜 (등급 S)

1. 체크포인트 — state.json `next_action`에 "사용자 피드백 대기: <요지>" 기록
2. journal.jsonl에 `{"event":"recovery_exhausted", ...}` 또는 S 판정 fail 기록
3. 사용자 보고: 실패 키·등급, 시도 횟수, journal 관련 라인, **제안 선택지 2개 이상**
4. 대기. 우회 가능한 다른 큐 단위가 있으면 해당 단위를 blocked로 표시하고 다음 단위는 진행 가능
5. 사용자 피드백 수신 → `{"event":"feedback", ...}`으로 journal에 기록 후 재개

## 계상 규칙 (실패 카운터)

- **재시도·회복 성공 여부와 무관하게, 발생한 모든 fail은 실패 키 카운터에 계상한다.**
- 실패 키 규격: `분류:하위유형` (예: `structural:상호참조-깨짐`). 새 키 생성 전 기존 키 재사용 우선.
- 카운터 갱신은 이 플레이북의 책임: fail 기록 시점에 `improve.fail_counts[키]` +1.
- 같은 사건의 등급 승격 라인(`"escalated":true`)은 재계상하지 않는다 — 계상은 사건당 1회.

## 반복 실패 → full 평가 신호

같은 실패 키 3회 누적은 mandatory event다. checker가 이를 `full`로 라우팅하며, full 평가가 하네스 결함을 확인한 경우에만 `loops/IMPROVE-LOOP.md`로 넘긴다. 이 플레이북은 개선을 직접 실행하지 않는다.
